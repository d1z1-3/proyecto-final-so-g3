Desarrollo de la Infraestructura

Componente 1: Virtualización con Linux

Despliegue de dos entornos virtuales independientes utilizando la distribución asignada (Fedora 44).

  • VM 1: Fedora 44 (Entorno Gráfico de Escritorio).
  • VM 2: Fedora 44 Server (Entorno de Consola Pura).
  • Particionamiento: Configuración manual del disco con las áreas requeridas: /, swap y /home.
  • Red: Enlace inter-VM mediante adaptador de red interna, habilitando el acceso remoto seguro vía SSH de la VM gráfica a la de consola.

Evidencias de Implementación:

Particionamiento manual y asignación de discos:

Prueba de conectividad inter-VM y acceso SSH funcional:

[dajo12@fedora ~]$ ssh usuario_servidor@ip_maquina_consola

Componente 2: Contenedores Docker

Aislamiento de servicios locales mediante la instalación y aprovisionamiento de Docker Engine nativo en la máquina virtual gráfica.

  • Frontend: Servidor web Nginx que aloja la estructura estática del sitio (Puerto 80) mapeado con volúmenes persistentes.
  • Backend: Servidor básico HTTP en Python configurado y expuesto en el puerto 5000.
  • Orquestación: Automatización del despliegue unificado y gestión de volúmenes persistentes con Docker Compose utilizando como imagen base fedora:44.

Evidencias de Implementación:

Despliegue exitoso de contenedores mediante Compose:

[dajo12@fedora ~]$ docker compose up -d

Componente 3: Orquestación con Kubernetes

Configuración del clúster local de un nodo mediante Minikube para la gestión automatizada de alta disponibilidad.

  • Manifiestos (/k8s): Definición lógica estructurada en archivos YAML:
    • backend-deployment.yaml — Configuración optimizada con 3 réplicas para el despliegue orquestado del backend.
    • frontend-nginx.yaml — Configuración del despliegue del frontend con Nginx expuesto mediante un servicio NodePort en el puerto 30080.
  • Réplicas: Despliegue inicial parametrizado y escalado manual verificado a 3 réplicas plenamente operativas para mitigar sobrecargas de tráfico.

Evidencias de Implementación:

Verificación del estado de los Pods y el Servicio expuesto:

[dajo12@fedora ~]$ kubectl get pods
[dajo12@fedora ~]$ kubectl get svc

Escalado de la infraestructura a 3 réplicas activas:

[dajo12@fedora ~]$ kubectl scale deployment/nginx-deployment --replicas=3

Conclusiones del Proyecto

  • El despliegue y escalado de réplicas en Kubernetes mediante Minikube demostró la robustez de la plataforma para mantener la alta disponibilidad del sistema. Al definir 3 réplicas en el manifiesto de despliegue, el clúster balancea automáticamente la carga de peticiones hacia los pods activos, garantizando tolerancia a fallos y estabilidad tanto para el backend en Python como para el servidor Nginx.
  • Inicialmente se presentaron conflictos de enrutamiento y solapamiento de puertos al intentar servir la documentación técnica en el mismo entorno de red local que los contenedores de la aplicación web. Aislar por completo el portal de documentación dentro del directorio /docs y delegar su despliegue de forma independiente a través de GitHub Pages solucionó el problema de raíz, permitiendo un acceso público y limpio sin interferir con el tráfico del clúster.
  • Para futuros proyectos a gran escala, se recomienda automatizar el flujo de despliegue mediante un pipeline de CI/CD (como GitHub Actions) que compile las imágenes de Docker ante cada cambio en el repositorio principal. Asimismo, es aconsejable implementar políticas de recursos (CPU y memoria limit/requests) en los manifiestos de Kubernetes para asegurar un uso eficiente del hardware del nodo.
